# 复杂任务指令

适用范围：所有被判定为“复杂任务（Complex）”的场景。

> 入口文件为 `./AGENTS.md`。当任务被判定为复杂任务时，必须先阅读并遵循本文件，再进入实现阶段。

## 复杂任务工作流

**适用范围**：所有被判定为复杂任务的场景。

### 1. 读（Research）

目标：理解现状，确认影响范围，避免重复造轮子。

检查清单（按需裁剪，但不要跳过高风险项）：

- [ ] 读取规格文档（如存在）：`SPEC.md`、`docs/spec/`、`docs/api/`、`.agent/SPEC.md`
- [ ] 读取项目根 `.agent/CONTEXT.md`（若存在）；并按路径自顶向下读取相关目录的 `.agent/CONTEXT.md`
- [ ] 阅读相关源码：调用方/被调用方、接口/类型定义、配置/环境变量、测试/示例、相关文档
- [ ] 阅读 ≥2 个相似实现或测试用例（若代码库中存在）
- [ ] 查阅官方文档确认正确用法（特别是框架/库的关键 API、配置、迁移指南）

产物：对问题的清晰理解 + 可复用组件列表 + 初步影响范围（会影响哪些模块/接口/数据/配置）。

### 2. 想（Plan）

目标：设计简洁方案，预见风险，明确验证与回滚。

检查清单：

- [ ] 数据结构优先：核心实体是什么？如何关联？边界条件是什么？
- [ ] 行为与兼容性：是否改变对外接口/行为？是否需要兼容旧数据/旧协议/旧配置？
- [ ] 风险点枚举：权限/安全边界、数据迁移、并发/性能、灰度/回滚
- [ ] 消除特殊案例：能否统一处理而非堆砌 if-else？
- [ ] 验证与回滚：有哪些可运行的验证步骤？回滚策略是什么？

产物：简要实施计划（阶段 + 验证点 + 回滚点），必要时拆分为可独立交付的阶段。

### 3. 做（Implement）

目标：高质量实现并记录。

检查清单：

- [ ] 实现变更并同步更新相关测试/示例/文档
- [ ] 运行可复现的验证步骤（单测/集成/手工验证，至少覆盖核心路径与边界）
- [ ] 完成后更新最近的 `.agent/CONTEXT.md`（见下文“项目记忆”）
- [ ] 若触发“任务过程追踪（TASK）”，创建并持续更新 `.agent/TASK-*.md`（见下文“任务过程追踪”）

## 任务过程追踪（TASK）

### 1. 何时需要创建 TASK 文档

**决策表**：满足以下条件时必须创建 `.agent/TASK-[任务名].md`：

| 预计实施阶段 | 涉及模块/目录 | 高风险点                   | 必须 TASK |
| ------------ | ------------- | -------------------------- | --------- |
| ≥3           | ≥3            | 任意                       | 是        |
| 1-2          | ≥3            | 数据迁移/安全边界/接口兼容 | 是        |
| ≥3           | 1-2           | 数据迁移/安全边界/接口兼容 | 是        |
| 1-2          | 1-2           | 无                         | 否        |

启用“子代理并行协作 / 双源复核”时，无条件创建。
用户明确要求记录时，无条件创建。

### 2. 必须记录的实施过程要素

TASK 文档必须包含并持续更新以下内容（可精简，但不可缺失）：

- 当前进度：已完成/进行中/待办步骤
- 关键决策与变更：为什么改、改了什么、影响范围
- 问题与解决：问题描述、尝试方案、最终方案
- 下一步行动：明确下一步是什么、依赖条件
- 残余风险/待验证项：当前无法覆盖/无法验证/仍有争议的点 + 下一步验证建议
- 自检-纠偏记录：关键不确定性/失败/范围变化时的复盘与纠偏动作（若触发）
- 并行协作记录：启用子代理或多角色复核时，记录分工、证据与采纳结论（若启用）

模板：`~/.agent/templates/TASK.md`

## 项目记忆（CONTEXT / CHANGELOG）

### 1. CONTEXT.md 结构（约定）

```
<project>/.agent/CONTEXT.md              # 项目级：全局约定、目录索引
<project>/.agent/CHANGELOG.md            # 变更历史：按需查阅的完整历史记录
<module>/.agent/CONTEXT.md               # 模块级：模块边界、关键接口
<subdir>/.agent/CONTEXT.md               # 目录级：具体实现约定
<project>/.agent/TASK-[任务名].md       # 任务级：单次任务记录（当需要任务过程追踪时）
```

模板：`~/.agent/templates/CONTEXT.md`

设计原则：

- 索引优先：CONTEXT.md 是导航，不是百科全书
- 详情外置：详细设计放到 `docs/` 或子模块 CONTEXT.md
- 按需更新：架构调整、全局约定变更时更新

Token 控制目标：

- 项目根：≤ 800 tokens
- 模块级：≤ 400 tokens
- 子目录级：≤ 200 tokens

### 2. 读取顺序（复杂任务）

1. 任务开始：读取项目根 `.agent/CONTEXT.md`（若存在）
2. 访问目录前：按路径自顶向下读取相关目录的 `.agent/CONTEXT.md`（若存在）
3. grep/glob 仅作辅助，不可替代项目记忆

若缺失 `.agent/CONTEXT.md`：优先通过源码阅读推进；仅在需要沉淀长期记忆/跨模块协作/架构调整时创建（保持轻量，索引优先）。

### 3. 更新要求（复杂任务）

完成修改后，必须更新最贴近的 `.agent/CONTEXT.md`，包含：

- 变更摘要：做了什么、为什么、何时
- 影响范围：改动的文件、依赖关系
- 验证步骤：如何测试、结果在哪
- 决策记录：关键选择及理由

### 4. 更新决策规则

- 单目录改动：更新该目录的 `.agent/CONTEXT.md`
- 跨目录改动：更新共同父目录的 `.agent/CONTEXT.md`
- 新增模块：创建模块的 `.agent/CONTEXT.md` + 更新项目根
- 架构性变更：更新项目根 `.agent/CONTEXT.md` + 追加到 `.agent/CHANGELOG.md`

CHANGELOG 模板：`~/.agent/templates/CHANGELOG.md`

## 质量标准与安全检查

### 1. 代码质量

- 数据结构优先于算法；消除特殊案例；不超过 3 层缩进（复杂函数需拆分）
- 清晰命名；避免不必要的抽象；优先复用项目内已有实现

### 2. 测试与验证

- 提供可运行的验证步骤（单元测试 / 集成测试 / 手工验证）
- 覆盖核心路径与边界情况
- 将验证步骤与结果记录到 TASK（若创建）或提交说明中

### 3. 安全检查

- 输入验证：防注入、防越界/溢出
- 权限控制：最小权限原则、边界清晰
- 数据保护：敏感信息加密/脱敏，避免日志泄露

## 工具策略（含 MCP）

### 1. 优先级（从高到低）

1. 项目记忆：`.agent/CONTEXT.md`（若存在且已启用）
2. 源码阅读：直接读取相关文件与测试
3. 官方文档：语言/框架/库官方文档
4. 开源实现：参考成熟项目的最佳实践
5. 搜索引擎：通用问题查询

### 2. MCP 工具选择

> 仅当“项目记忆 + 源码阅读 + 官方文档”不足以解决问题时，才考虑使用 MCP。

| 任务意图                 | 主要服务        | 备用   | 使用时机                                     |
| :----------------------- | :-------------- | :----- | :------------------------------------------- |
| 简单 bugfix / 小范围改动 | 直接阅读 + 编辑 | 无     | 单文件、小范围改动，逻辑清晰，本地上下文充足 |
| 官方文档/API/框架        | `context7`      | `curl` | 库用法、版本差异、配置问题                   |

### 3. 使用前置条件与降级策略

- `context7`：仅针对公共库/框架/API；若问题涉及项目内自定义模块，优先查看本地代码与 CONTEXT，再考虑调用
- 若某 MCP 多次调用失败，应切回“项目记忆 + 源码阅读 + 备用工具”，不要在同一工具上反复重试

### 4. 子代理并行协作（Complex 强制规则）

用途：并行检索/独立复核/对手质询/测试矩阵等“只读”工作，降低漏项与拍脑袋决策风险。

触发条件（满足任一时建议启用；高风险项必须启用）：

- 可并行子问题 ≥ 2（如：上下文扫描、多模块检索、外部事实核对、对手质询、测试矩阵等）
- 高风险决策需独立复核（安全/权限/删除/数据迁移/破坏性脚本等）
- 跨领域且单链路易漏项

执行规则：

- 并行度：默认 3-5，硬上限 8（超出需在 TASK 记录原因与预期收益）
- 职责分工：子代理只做只读工作（读文件/跑只读命令/检索/推导/给出备选方案）；所有写操作（尤其 `apply_patch`、移动/删除等破坏性动作）仅允许主代理执行
- 双源复核：高影响结论必须满足其一：
  - ≥2 个独立证据来源一致（例如：不同文件/测试输出/官方文档 vs 代码行为），或
  - ≥2 个子代理独立推导结果一致
- 日志留痕：每个子代理在 TASK 的“并行协作记录”里记录 `agent_id/角色`、分工、关键输出摘要、证据与采纳/否决理由
- 收尾流程：若平台支持子代理生命周期管理，则子代理结束前输出 `READY_TO_CLOSE`，由主代理关闭；若不支持，则在 TASK 记录“平台能力限制”并以等价方式完成（例如：主代理汇总子代理输出并标记已完成）

### 5. 子代理路由（可选配置）

如果项目有 `.agent/router.yaml`，按配置路由：

```yaml
routes:
  "*.go": go-expert
  "*.py": python-expert
  "*.{js,ts}": frontend-developer
  default: general-purpose
```

## 异常处理

### 1. 任务暂停

- 若已创建 TASK：将状态标记为“已暂停”
- 在“实施过程”中记录暂停原因与已完成进度
- 在“下一步行动”中明确恢复条件与所需资源

### 2. 方案变更

- 在“关键决策”中新增条目，记录方案变更
- 说明原方案不可行原因（技术限制/性能问题/发现更优方案）
- 描述新方案的技术路径与选择理由
- 做影响评估：对已完成工作的影响，是否需要回退

### 3. 任务取消

- 若已创建 TASK：将状态标记为“已取消”
- 记录取消原因（需求变更/优先级调整/技术不可行）
- 经验总结：避免将来重复
- 清理工作：如有部分改动，说明回滚步骤或遗留状态
