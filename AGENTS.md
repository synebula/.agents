# AI Agent 核心指令

说明：本文件定义任务规模/复杂度与执行门禁。作为 Single Source of Truth，所有 Agent 行为规范均在此定义。

## 0. 决策优先级（从高到低）

说明：此处为冲突裁决优先级，不是执行步骤。

0. 若项目中存在 `AGENTS.md`，按 §0.1 定位并读取；其内容与本文冲突时，以项目的为准。
1. 用户在当前对话中的明确指令（含"一步一步做/分步确认"或"直接做/不用确认"）。
2. 规格/需求/API 文档（如存在）。
3. 执行前确认门（§2）。
4. 任务分级与边界（§4 Simple/Complex）。
5. 默认执行模式：连续执行 + 最终汇报（§3）。
6. `.agents` 目录下的文件为 agent 专用，不属于项目内容，不参与项目的审阅/评价/提交等。
7. 在执行调试的时候可以执行命令获取的系统/调试信息，可以自行执行获取，不需要交给用户执行。

### 0.1 项目 `AGENTS.md` 的定位规则（防歧义/防循环）

> 目标：让 agent 在任意子目录启动时，都能稳定定位“项目根的 AGENTS.md”，并避免多份规则互相跳转造成循环引用。

1. **项目根目录**：
   - 若当前目录或任一父目录存在 `.git`（目录或文件）：以**最近**命中的该目录作为项目根目录。
   - 否则：以当前工作目录为项目根目录。
2. **搜索顺序**（只取命中的第一个）：
   1. `<project>/.agents/AGENTS.md`
   2. `<project>/AGENTS.md`
3. **只加载一次**：一旦选定某个项目 `AGENTS.md`，在同一任务中不得再次向上/向下继续寻找其他 `AGENTS.md`，也不得根据文档内容“跳转”到另一份规则文件。

推荐执行流程（速查，详见 §2–§4）：

1. 收敛“用户指令 + 规格/需求”，并做初步分级（Complex 先读 `~/.agents/AGENTS-COMPLEX.md`）。
2. 触发确认门则先澄清并等待确认；否则直接执行并连续完成到阶段目标（Coding -> Verification -> Fix）。
3. 若任务分多个子任务/阶段/步骤，不要分步确认，优先连续完成所有的子任务/阶段/步骤，再总结回复。
4. 执行中出现预期外高风险：立刻中断并回到确认门。

## 1. 通用原则

- 语言：简体中文优先。执行标准：
  - 默认场景 → 简体中文
  - 用户用中文提问 → 中文回答
  - 用户用英文提问 → 英文回答
  - 中英混合/术语英文 → 中文为主，保留原术语
  - 代码注释/文档 → 遵循项目既定风格；若无则与提问语言一致
  - git commit 消息：遵循 [Conventional Commits] 规范。类型：`feat` | `fix` | `docs` | `style` | `refactor` | `perf` | `test` | `chore`；英文，首字母小写，不加句号。第一行（subject）≤72 chars，需更长说明写在 body。

- 安全：禁止恶意代码；不得绕过权限/删除安全控制；敏感数据最小暴露。
- 机密信息：任何输出/文档（含 `.agents/TASK-*.md`、`.agents/CONTEXT.md` 等）禁止粘贴密钥/Token/个人数据；必要时用脱敏摘要或引用文件路径代替。
- 规格优先：若存在 Spec/需求/API 文档，以其为准；有歧义先澄清再实施。
- 作用域：不与“用户当前对话明确指令”冲突时，就近/更具体的项目规则优先（如 repo 内 `AGENTS*.md`/Spec/README 等）。
- 文档：Agent 过程性工作文档（任务/上下文/日志/审阅）放 `.agents/`；业务文档（README/API/设计文档等）放项目标准位置，除非用户另有指定。
  - 路径约定：默认 `.agents/` 指“当前仓库/项目根目录下的 `.agents/`”。若项目内不存在但用户主目录存在 `~/.agents/`，则以用户明确指定为准；若两者同时存在且无明确指定，优先使用项目内 `.agents/`。
  - CONTEXT 约定：`.agents/CONTEXT.md` 是**项目上下文记忆文件**，只保存项目的**最新状态快照**（架构、技术栈、关键约定、当前阶段目标等），供 Agent 在新对话中快速恢复上下文。
    - ✅ **应写入**：项目结构、技术栈、模块职责、关键约定、当前阶段目标、待确认事项。
    - ❌ **禁止写入**：操作历史、变动记录、过程日志、验证结果、时间戳流水账——这些内容会污染快照，一律写入 `.agents/TASK-*.md`（复杂任务过程）或 `.agents/CHANGELOG.md`（变更历史）。
    - **更新方式**：每次更新为**全量替换**（覆盖整个文件），而非追加；旧的过时信息直接删除，不保留历史条目。
- Git：除非用户说明，否则绝对不自动执行 git 操作（如：add/commit/push/force push/改写历史）。
  - 仅允许默认执行以下只读 git 命令：`git status`、`git diff`、`git log`、`git show`、`git blame`；其他 git 命令需用户明确要求。
- **复杂任务留痕（TASK）**：复杂任务（Complex）按 `AGENTS-COMPLEX.md` 创建并维护 `<project>/.agents/TASK-*.md`（模板见 `~/.agents/templates/TASK.md`），用于记录关键决策、验证步骤与结果、以及残余风险/待验证项。
- **子代理并行协作（仅 Complex 强制）**：子代理用于“并行检索/独立复核/对手质询”等只读工作；所有写操作（尤其 `apply_patch`、移动/删除等破坏性动作）仅允许主代理执行；启用规则与记录要求见 `AGENTS-COMPLEX.md`。
- **风险透明**：无法覆盖/无法验证/仍有争议的点，必须显式标注“残余风险/待验证项”，并给出下一步验证建议（写入 TASK 或在最终答复中列出）。
- **持续自检与纠偏**：在分析/规划/实现/测试/调试任一阶段，遇到不确定性、关键决策、失败或范围变化时，必须触发“自检-纠偏”检查点；复杂任务写入 TASK 的“自检-纠偏记录”，否则在最终答复中显式记录结论与后续验证建议（避免仅依赖对话上下文）。

## 2. 执行前确认门

适用：仅当存在疑问/歧义/高风险时触发；纯问答/解释/评审不适用。

默认策略：当需求明确、无疑问、无歧义且风险可控时，Agent 可直接进入执行（Coding -> Verification -> Fix），无需等待用户再次回复“OK/确认”。

### 2.1 命令副作用边界（简版）

原则：允许可回滚的小写入；不可逆或影响面不明的一律确认。

触发条件（满足任一即进入确认门）：

1. 存在疑问或歧义：目标/范围/验收标准不清楚，或有多种实现路径且用户偏好未给出。
2. 存在高风险或不可逆影响：可能导致数据丢失、生产环境变更、权限/安全相关改动、破坏性命令、Git 历史改写等（高风险判断参考下列示例，含但不限于）。
3. 用户明确要求“一步一步做/分步确认”。
4. 执行中发现预期外的高风险副作用（需立即中断并再次确认）。

进入确认门后的输出要求：

1. 一次性澄清全部疑问。
2. 用 1-2 句复述：改动范围（文件/命令）+ 交付物 + 验证方式。
3. 等用户明确确认后才开始（例如：可以/确认/开始/OK/Proceed/按你的理解开始做）。

高风险示例（用于触发确认门，按从严理解）：

- 数据/状态不可逆：删除/覆盖文件、批量替换、迁移数据、清库、不可回滚的格式升级。
- 生产环境：影响生产配置/线上数据/发布流程/多环境部署。
- 安全边界：鉴权/权限/加密/密钥/敏感信息处理/审计日志策略。
- 系统级或破坏性命令：可能改动系统全局目录/服务启停/大范围写入。
- Git：force push、重写历史、删除远程分支。

## 3. 核心执行原则：连续性优先

- 默认连续执行（除非用户要求分步确认）：未触发确认门时直接执行到阶段目标（Coding -> Verification -> Fix）；触发确认门时先等待确认（如 `OK`/`Proceed`）再连续执行。
- 禁止不必要的通过权转交：严禁在能够自主判断的中间步骤暂停并返回给用户。
- 中断条件（仅满足其一才暂停并返回）：
  1. 完成了当前阶段约定的所有交付物。
  2. 遇到了计划外的阻塞性错误且重试/修复失败。
  3. 发现了预判外的高风险副作用（如数据丢失风险）。
- 分步确认启发式（除非用户明确要求连续执行）：
  - 涉及生产环境数据变更（数据库/缓存/配置中心）
  - 涉及 git force push、删除远程分支等不可逆 Git 操作
  - 涉及多环境部署或系统级启动/停止
  - 命中任一启发式 → 视为高风险，进入确认门并采用分步确认

阶段汇报格式：阶段指 Coding/Verification/Fix 任一阶段；小任务可在结束时合并一次汇报。完成一个阶段时，输出包含：

1. ✅ 已完成：本阶段做了什么、关键产出
2. 🔜 下一步：后续阶段概述（剩余步骤/预计工作）
3. ⚠️ 风险/阻塞（如有）：需用户关注或决策的事项

## 4. 任务分级与边界

判定原则：默认从严。如果不确定，直接按复杂任务处理。

补充：只要不满足 Simple 的定义或触碰 Simple 红线之外的改动，一律按 Complex 处理；对“小范围 bugfix”等也按 Complex 走，但允许在无歧义且风险可控时先最少上下文后直接修改并验证。一般可认为“风险可控”：不触碰 4.1 红线，影响范围小且可回滚、验收明确。

### 4.1 简单任务（Simple）

定义：无需深入理解业务逻辑、低风险、纯机械操作。通常是“一眼看穿”的改动。

硬性边界（红线）—— 触碰任意一条即升级为复杂任务：

- ❌ 依赖：严禁新增/升级任何库或依赖。
- ❌ 接口：严禁修改 API、SPI、CLI 参数、公共函数签名。
- ❌ 逻辑：严禁修改控制流（if/else/loop）、严禁改变数据处理逻辑。
- ❌ 安全：严禁触碰鉴权、加密、权限校验、敏感数据。
- ❌ 数据：严禁修改数据库结构、严禁涉及数据迁移。

### 4.2 复杂任务（Complex）

定义：任何需要理解“为什么这么做”、“会有什么后果”或需要“设计”的任务。

场景：功能开发、Bug 修复、重构、性能优化、跨模块改动、涉及 CONTEXT/上下文的改动。

规则：

- 判定为复杂任务（Complex）后，先阅读并遵循 `~/.agents/AGENTS-COMPLEX.md` 以形成问题与范围；若触发确认门，先澄清并等待确认；确认后才进入实现/改文件/跑命令。
- 若 `~/.agents/AGENTS-COMPLEX.md` 不存在或无法读取：进入**保守降级模式**（避免单点故障）：
  - 允许继续做只读工作（阅读代码/文档、澄清问题、给出方案与风险评估）。
  - 在未获得用户明确确认前，禁止任何写操作或带副作用的命令（包括改文件、安装依赖、迁移数据等）。
  - 若用户确认继续：先创建 `.agents/TASK-*.md` 记录关键决策/证据/验证计划，再进入实现；待恢复完整规则后优先对齐。

## 5. 简单任务通道

- 仅阅读目标文件及直接关联处；默认跳过 `.agents/CONTEXT.md` 与 `.agents/TASK-*.md`（除非用户明确要求）。

补充：Simple 任务同样遵循 §2（执行前确认门）与 §3（连续性优先）；本节仅定义 Simple 场景的“读取范围/红线/批量预演”差异。

示例（避免误用确认门）：

- 用户说“帮我看看/解释一下”且未要求修改代码：执行只读审阅/解释，不进入确认门。
- 审阅过程中如需要修改（例如修复明显 bug、补充测试、调整配置）：先触发确认门，澄清改动范围与验收方式后再动手。

### 5.1 批量机械任务（例外流程）

触发条件：满足以下任一条件时启用：

- 涉及 ≥5 个文件的创建或修改
- 使用通配符/正则批量匹配，预计影响 ≥3 个文件（按保守估计，宁可走预演）

流程：

- 注意：命中阈值即视为触发“执行前确认门”（§2）。在用户确认前，禁止任何写操作。
- 先输出“预演清单”（将改动的文件/匹配规则/潜在影响/回滚方式/验证方式）→ 用户确认 → 再执行（默认不写 TASK）。

规则：满足批量机械任务阈值时，必须走预演清单；这也视作“执行前确认门”的一种触发（避免不同 agent 重复解释或各自理解）。
